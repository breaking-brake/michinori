## デモ②（1:30–2:20）／ `/chat` でDAGを育てる

**シナリオ**: デモ①のDAGに「もう終わってる作業ある？」と相談 → AIがコードを調べて実装済み部分を発見 → ステータス更新を提案 → 承認で反映

| 時間 | 画面・操作 | テロップ | 演出ノート |
|---|---|---|---|
| 1:30–1:38 | チャットパネルを開く → 質問入力 | 「この中で、**コード的にもう終わってる作業**ある？」 | デモ①のDAGを背景に残して連続性を出す |
| 1:38–1:52 | 送信 → AIが調査中の表示 | 「AIが**実際のコードを調査中**…」<br>（可能なら）「terraform / Dockerfile / CI を確認」 | ★下記「要確認」参照。ツール実行が出るなら見せる／出ないならテロップで担保 |
| 1:52–2:05 | AI回答テキスト＋**提案カード**が出現（reasoning付き） | 「“推測”じゃなく、**実物を見て**判断」<br>（回答例）「デプロイ設定とCIは既にコードあり → 進行中に更新を提案」 | 回答は短く。reasoningが1〜2行見えると説得力大 |
| 2:05–2:12 | 「DAGに反映」クリック → **プレビューダイアログ** | 「提案は、**あなたの承認で**反映」 | 2段階承認の1段目＝“勝手に壊さない安全設計”の見せ場 |
| 2:12–2:17 | プレビューを一瞬見せ →「確定」 → **ノード色が変化** | 「ワンクリックで、計画が最新に」 | ステータス色が視覚的に変わる瞬間を捉える |
| 2:17–2:20 | 更新後のDAG全体に引く → 締めへ | 「確認も反映も、**最小の手数で**。」 | 次パートへの橋渡し。“ずれない”ではなく手数の少なさを強調（現状の実力に即した表現） |

**設計意図**
- 「コードを読む」を映像で証明（デモ①の「人がゴールを言うだけ」と対になる）
- 「提案→承認」の安全設計を明示（AIが勝手に壊さない不安を先回りで解消）
- 色が変わる瞬間で“効いてる感”を体感

**★要確認（リハ時にチェック）**
チャットのツール実行（`search_code`/`read_file`）が **UI上に途中経過として表示されるか**。実装上 `/chat` はサーバー内で最大8ラウンドのツール呼び出しを回してから結果を返すため、フロントに中間ステップが出ない可能性がある。
- 出る場合 → 1:38–1:52 でその調査ログを見せる
- 出ない場合 → ローディング中テロップ「AIが実際のコードを調査中…」で補い、回答の reasoning で“コードを見た証拠”を示す
