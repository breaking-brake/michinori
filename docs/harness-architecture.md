# Michinori ハーネス設計

## 概要

Michinoriは、Gemini 2.5 Flashをエージェントとして使い、プロジェクトDAGの計画・修正を行うハーネスアーキテクチャを採用している。

- **エージェント:** Gemini 2.5 Flash（tool-use / function calling）
- **ハーネス:** Cloud Run Server + Client UI
- **承認ゲート:** ユーザーの明示的な確定操作なしにDAGは変更されない

## エージェント設計

### Geminiの2つのモード

| エンドポイント | 用途 | Geminiの使い方 |
|---|---|---|
| `POST /analyze` | 初回DAG生成 | 構造化出力（responseSchema） |
| `POST /chat` | DAG修正・相談 | function calling（tool-use, mode: AUTO） |

### ツール定義: propose_dag_changes

Geminiが「ユーザーの変更意図が明確」と判断した場合に呼び出すツール。

パラメータ:
- `reasoning` (string): 変更理由（日本語）
- `proposal_json` (string): 変更内容のJSON文字列

proposal_jsonの構造:
```json
{
  "additions": [{
    "id": "kebab-id",
    "label": "タスク名",
    "description": "説明",
    "estimateMd": 1.0,
    "category": "実装",
    "status": "未着手",
    "dependencies": [],
    "position": { "x": 0, "y": 0 }
  }],
  "removals": ["node-id-to-remove"],
  "modifications": [{
    "nodeId": "existing-id",
    "changes": {
      "label": "新名",
      "estimateMd": 2.0,
      "position": { "x": 100, "y": 200 }
    }
  }]
}
```

注: positionはユーザーが明示的にレイアウト調整を求めた場合のみ含める。通常はUIの自動レイアウトに任せる。

### AIの自律判断

`FunctionCallingConfigMode.AUTO`を使用し、Geminiが各メッセージに対して自律的に判断:

- **テキスト応答:** 質問への回答、確認、議論
- **ツール呼び出し:** 変更意図が明確な場合にDAG変更を提案

ハーネス（サーバー）はツール呼び出しを検出し、proposalとして構造化してクライアントに返す。

## 承認フロー

```
ユーザー発言
    ↓
Gemini判断（AUTO）
    ├─ テキスト応答 → チャットに表示
    └─ propose_dag_changes → 提案ブロック表示
                                ↓
                    入力エリアが切り替わる
                    [DAGに反映] [キャンセル]
                        │           │
                        │           └→ 提案をdismissed
                        │              入力エリア復帰
                        │              会話継続
                        ↓
                    プレビューダイアログ
                    (変更後DAGをReact Flowで表示)
                    [確定] [閉じる]
                        │       │
                        │       └→ プレビューを閉じる
                        │          提案はまだ有効
                        ↓
                    DAGに適用
                    criticalPath再計算
                    永続化
```

## 会話管理

- **ステートレス:** サーバーは会話状態を持たない
- **クライアント管理:** 会話履歴はクライアント側で保持し、毎リクエストで送信
- **コンテキスト:** システムプロンプトに現在のDAG全体を含める
